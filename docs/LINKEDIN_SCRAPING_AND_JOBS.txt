LINKEDIN IMPORT (RELEVANCE AI) AND REMOTE JOBS — STEP-BY-STEP
===============================================================

This file describes how the app pulls LinkedIn profile data and aggregates job
listings. It maps each step to the code that implements it.

WHY “WEB SCRAPING” OFTEN FAILS (READ THIS FIRST)
-----------------------------------------------
LinkedIn is not designed to allow anonymous server-side HTML scraping:

• HTTP 999 — LinkedIn’s custom “Request denied” for suspected bots (very common
  from cloud/datacenter IPs and from Node `fetch` without a real browser session).

• Login / authwall / challenge HTML — You get a sign-in or checkpoint page
  instead of public profile markup, so JSON-LD and Open Graph tags are missing.

• Profile must be public — Private profiles cannot be read without OAuth.

• Relevance AI runs *before* the HTML scraper in `server/index.ts`. If the
  webhook returns usable profile JSON, scraping is skipped. If Relevance fails
  or returns an error, the app falls back to `scrapeLinkedInProfile` in
  `server/linkedinScrape.ts`, which is where most “it doesn’t work” reports come from.

What to do: configure `RELEVANCEAI_WEBHOOK_URL` / `RELEVANCEAI_API_KEY`, use
**Resume → profile (AI)** on Account, or paste experience manually. Do not expect
reliable long-term scraping without official API access.


PART A — ENVIRONMENT
--------------------
Step 1. Copy `.env.example` to `.env.local` (gitignored).

Step 2. Optional: set `RELEVANCEAI_API_KEY` (server only; no `VITE_` prefix).
If set, it is sent as the `Authorization` header on the webhook POST. If unset,
the request is still sent (some public triggers work without a key).

Step 3. Optional: set `RELEVANCEAI_WEBHOOK_URL` to override the default. The
shareable form URL (`app.relevanceai.com/form/...`) is **not** the API endpoint;
copy the exact `trigger_webhook` URL from Relevance → your tool → Triggers
(it includes studio id `4b51cdf1-f328-477e-a4c4-b3760406073a` for this project).

Functions: none (configuration only).


PART B — LINKEDIN PROFILE IMPORT (CLIENT → SERVER)
--------------------------------------------------
Step 1. The UI (e.g. Account / Onboarding) collects a LinkedIn profile URL and
calls `importLinkedInProfile` in `src/services/linkedin.ts`.

Step 2. That function POSTs JSON `{ profileUrl }` to `POST /api/linkedin-import`
(same origin in dev when Express + Vite run together).

Step 3. `server/index.ts` validates the body, then calls `normalizeLinkedInUrl` from
`linkedinScrape.ts` so only well-formed public-style `/in/...` URLs proceed.

Step 4. The server calls `importLinkedInViaRelevance` in `relevanceLinkedIn.ts`.

Functions:
- `src/services/linkedin.ts` — HTTP client for `/api/linkedin-import`
- `normalizeLinkedInUrl` — `linkedinScrape.ts`
- Route handler — `server/index.ts` (`POST /api/linkedin-import`)


PART C — RELEVANCE AI (SYNC WEBHOOK VS ASYNC)
---------------------------------------------
• **Synchronous** `trigger_webhook`: POST body `{"url","name"}`. Times out around
  **30 seconds** — fine for fast tools only.

• **Asynchronous** `trigger_async` + `async_poll`: implemented in
  `importLinkedInViaRelevanceAsync` in `relevanceLinkedIn.ts`. Enabled when
  `RELEVANCEAI_REGION`, `RELEVANCEAI_TOOL_ID`, and `RELEVANCEAI_PROJECT_ID` are
  all set (see `.env.example`). Authorization header is
  `project_id:api_key` from the API Keys page (or `RELEVANCEAI_AUTHORIZATION`).

• Async POST body: `{"params":{"url":"<linkedin>","name":""},"project":"<project_id>"}`.
  Your Relevance tool must accept those **param** names (or change the tool’s
  input schema to match).

• Set `RELEVANCEAI_USE_ASYNC=false` to force the sync webhook even if async
  env vars are present.


PART D — RELEVANCE AI WEBHOOK REQUEST (SYNC)
--------------------------------------------
TROUBLESHOOTING (if import shows a red “Relevance failed” message or empty data):

• You MUST set `RELEVANCEAI_WEBHOOK_URL` to **your** Relevance project’s
  `trigger_webhook` URL. The hard-coded `DEFAULT_WEBHOOK` in
  `server/relevanceLinkedIn.ts` is only a demo and may return 401/404 or an
  empty payload for your account.

• After changing `.env.local`, restart `npm run dev` (server reads env at startup).

• If Relevance returns HTTP 401/403, check `RELEVANCEAI_API_KEY` (copy exactly
  from Relevance; add `Bearer ` prefix only if their UI shows that format).

• If Relevance returns **HTTP 405** on a correct POST: the HTTP client may have
  followed a redirect with **GET** (webhooks are POST-only). This codebase uses
  `postTriggerWebhook` in `relevanceLinkedIn.ts` to follow `Location` while
  repeating POST with the same JSON body.

• **30-second synchronous limit (Relevance UI):** “Synchronous triggers” only
  wait ~30s. If your tool chain (scrape + LLM) runs longer, the webhook can
  return a timeout or incomplete body → “Could not parse profile fields.” Fix:
  shorten the tool, split steps, or use Relevance’s **asynchronous** trigger API
  (see their docs). This app’s server expects a **complete JSON body** on the
  same HTTP response.

• If HTTP 200 but “Could not parse profile fields”, your agent’s output JSON
  does not match what `mapRelevanceOutputToProfile` expects (nested
  `output`/`result`/`data` with `fullName` or `experience`/`skills` arrays).
  Inspect the server console: we log the first ~500 chars of the response.

• **Minimal shapes that work** (inside `output` / top-level, or under `result`/`data`):
  - At least one of: `fullName` or `full_name` or `name`, **or** `headline`, **or**
    `experience` / `experiences` array, **or** non-empty `skills` array.
  Example:
  `{"fullName":"Jane Doe","headline":"Student","skills":["Python","SQL"],"experience":[{"title":"Intern","company":"Acme","description":"..."}]}`

• The UI now surfaces `relevanceError` + scrape `error` in `importHint` when
  the server responds with 422.

Step 1. `importLinkedInViaRelevance` POSTs to the webhook URL. If
`RELEVANCEAI_API_KEY` is set, it adds `Authorization: <key>`; otherwise the
header is omitted.

Step 2. The JSON body matches the webhook contract: `{"url":"<linkedin profile
url>","name":""}` (`name` is optional).

Step 3. The raw response is parsed as JSON when possible; otherwise it is
wrapped as `{ raw: string }`.

Step 4. `getProfileRecord` unwraps common envelopes (`output`, `result`,
`data`). `mapRelevanceOutputToProfile` maps tool output into `UserProfile`,
including LinkedIn-style fields: `full_name`, `first_name`/`last_name`,
`experiences[]` (title, company, date_range, skills per row), `educations[]`,
`profile_image_url`, `linkedin_url`.

Functions:
- `importLinkedInViaRelevance`, `getProfileRecord`, `mapRelevanceOutputToProfile`
  — `relevanceLinkedIn.ts`


PART E — PRIVATE PROFILE DETECTION
----------------------------------
Step 1. After a successful HTTP response from Relevance, `detectPrivateProfile`
scans the full response text and recursively collected string fields from the
JSON for phrases like "private profile", "authwall", "sign in to LinkedIn",
etc.

Step 2. If a private/restricted pattern matches, the server responds with
HTTP 403 and `{ error, private: true }`. The client can show a clear message
that the profile must be public.

Step 3. If Relevance returns HTTP error status, the import returns an error
string (mapped to client behavior in `linkedin.ts`).

Functions:
- `detectPrivateProfile` — `relevanceLinkedIn.ts`
- `collectStrings` / `textFromPayload` — helpers in `relevanceLinkedIn.ts`


PART F — MAPPING RELEVANCE OUTPUT TO APP PROFILE
------------------------------------------------
Step 1. `mapRelevanceOutputToProfile` accepts arbitrary JSON and looks for
common shapes: `output`, `result`, or `data` wrappers, then fields such as
`fullName`, `headline`, `experience` / `work_experience`, `skills`.

Step 2. Experience entries are normalized to `Experience` objects with generated
ids via `rid()`.

Step 3. If structured experience is empty but a headline exists, a single
synthetic experience row may be built from headline + about/summary text.

Step 4. If no usable name, experience, or skills are found, the import fails
with a parse error (caller may fall back — see Part F).

Functions:
- `mapRelevanceOutputToProfile` — `relevanceLinkedIn.ts`
- `pickString`, `asRecord` — `relevanceLinkedIn.ts`


PART G — FALLBACK SCRAPE (WHEN RELEVANCE DOES NOT SUCCEED)
----------------------------------------------------------
Step 1. If Relevance does not return a full success (e.g. missing API key,
HTTP error, or empty parse), `server.ts` calls `scrapeLinkedInProfile` from
`linkedinScrape.ts`.

Step 2. Results are converted with `scrapeToUserProfile` into `Partial<UserProfile>`.

Step 3. On scrape failure, the server may respond with 422 and URL-only
fallback fields so the UI can still save the link.

Functions:
- `scrapeLinkedInProfile`, `normalizeLinkedInUrl` — `linkedinScrape.ts`
- `scrapeToUserProfile` — inline in `server.ts`


PART G — JOB LISTINGS (`GET /api/jobs`, `POST /api/jobs/personalized`)
----------------------------------------------------------------------
Step 1. Home / Search use `src/services/jobsApi.ts` `fetchJobsForUser(profile)`:
  signed-in users with profile data → `POST /api/jobs/personalized` (Gemini +
  heuristics build Adzuna queries); otherwise → `GET /api/jobs`.

Step 2. `server/index.ts` calls `aggregateRemoteJobs()` (optionally with
`adzunaWhatQueries` from personalization).

Step 3. `aggregateRemoteJobs` runs in parallel:
  - `fetchAdzunaJobs` — primary live feed when `ADZUNA_APP_ID` + `ADZUNA_APP_KEY`
    are set (see `.env.example`).
  - `fetchIndeedJobs` / `fetchLinkedInRssJobs` — **often return nothing** from a
    server: Indeed **403**, LinkedIn **404** or TLS drop (HTML error pages, not
    RSS). See PART H.
  - `fetchGoogleCareersJobs` — public JSON when available.
  - `handshakeSamples` — Handshake-labeled rows from `src/data/mockJobs.ts`.

Step 4. Results are merged (Google + Handshake samples before Adzuna so
deduplication keeps official Google rows when the same job appears twice),
deduplicated by `title|company`, and capped (`JOBS_AGGREGATE_MAX`).

Functions:
- `aggregateRemoteJobs`, `fetchAdzunaJobs`, `fetchIndeedJobs`,
  `fetchLinkedInRssJobs`, `fetchGoogleCareersJobs`, `handshakeSamples` —
  `jobsAggregator.ts`
- `resolveAdzunaQueriesForProfile` — `server/jobSearchPersonalization.ts`


PART H — FAQ: WHY LINKEDIN “IMPORT” FAILS; WHY NOT JOBS FROM INDEED/LINKEDIN
----------------------------------------------------------------------------
Two different features — do not confuse them.

A) LinkedIn **profile import** (paste your `/in/...` URL to fill name, skills,
   experience in the app)

   • LinkedIn does not offer a public “profile JSON API” for third-party apps
     the way many APIs do. Server-side **HTML scraping** is blocked (HTTP 999,
     login walls, private profiles).

   • This app’s supported path is **Relevance AI** (`RELEVANCEAI_*` in
     `.env.local`): your tool returns structured profile data; see Parts B–F.
     Use **async** (`trigger_async`) if the tool runs longer than ~30s.

   • If Relevance is misconfigured, the server falls back to `linkedinScrape.ts`,
     which usually **fails** for the reasons above. Alternatives: **Resume →
     profile (AI)** on Account, or enter experience manually.

B) **Jobs** from Indeed and LinkedIn (listings in Home / Search)

   • The code tries **public RSS-style URLs** in `jobsAggregator.ts`. From a
     datacenter or Node `fetch`, Indeed commonly returns **403 Forbidden** and
     LinkedIn **404** or connection errors — you get HTML, not job XML. That is
     **expected**; it is not fixed by firewall rules on your PC.

   • **Indeed** and **LinkedIn** job data for production apps normally requires
     **partner / commercial APIs** (application, compliance, fees). There is
     no stable “free API key” line in `.env.local` for this repo’s stack.

   • **What to do in this project:** set **Adzuna** keys (`ADZUNA_APP_ID`,
     `ADZUNA_APP_KEY`) and optionally `ADZUNA_QUERY_WHAT` for non-personalized
     fallback. Personalized queries use **Gemini** + profile (`POST
     `/api/jobs/personalized`). Many listings may still originate from major
     boards but come through Adzuna’s licensed feed, not direct RSS.

   • **Google Careers:** this app does **not** use “SERP” (search-engine result
     APIs like SerpAPI or Google Programmable Search). It calls Google’s public
     **`careers.google.com/api/v3/search/`** JSON endpoint (`fetchGoogleCareersJobs`
     in `jobsAggregator.ts`). That endpoint may return **empty** or fail from some
     server IPs (similar to RSS). Results are **deduped** by `title|company`; if
     Adzuna already listed the same Google role, the merge order prefers **Google +
     Handshake samples before Adzuna** so the `source: "Google"` row is kept when
     both exist.

   • To integrate **official** Indeed or LinkedIn job APIs later, you would add
     new server modules under their developer programs — separate from Adzuna.


PART I — PDF DOCUMENTS (SEPARATE FROM LINKEDIN)
-----------------------------------------------
Step 1. Document upload uses a file input accepting PDF; the browser runs
`extractPdfText` (`src/utils/extractPdfText.ts`) using pdf.js to extract text.

Step 2. Extracted text (not the binary) is what gets stored with the document
record.

Functions:
- `extractPdfText` — `src/utils/extractPdfText.ts`
- UI — `src/pages/Documents.tsx`
